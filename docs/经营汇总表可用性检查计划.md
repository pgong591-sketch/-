# 经营汇总表可用性检查计划

更新时间：2026-06-17

## 一、当前结论

经营汇总表现在没有数据显示，首要原因是“收入成本费用表”的分类汇总结果没有入库。

当前数据库检查结果：

| 检查项 | 结果 |
| --- | --- |
| 数据库文件 | `data/finance_dw.db` 存在 |
| `pl_detail` 行数 | 0 |
| `income_statement` 行数 | 1520 |
| `income_statement` 期间 | 202602、202603 |
| 已找到的收入成本费用表文件 | `data/incoming/202603_K051_income_cost_expense.xls` |
| 经营汇总表新口径 | 收入成本费用表分类汇总公式结果 |

所以页面没有数据，优先不是页面样式问题，而是收入成本费用表里的分类汇总区还没有被专门解析并落库。

## 二、检查目标

确认经营汇总表是否可用，要分 4 层判断：

1. 收入成本费用表文件是否存在。
2. 分类汇总公式区是否能识别。
3. 分类汇总结果是否已落库。
4. 期间和公司筛选是否能命中数据。
5. 页面是否能把结果展示出来。

只要前一层失败，后一层不要继续误判。

## 三、检查步骤

### 1. 检查收入成本费用表文件是否存在

当前已找到：

```text
data/incoming/202603_K051_income_cost_expense.xls
```

判断：

- 文件存在：进入解析检查。
- 文件不存在：需要先找回或重新上传收入成本费用表。

### 2. 检查分类汇总区是否能识别

收入成本费用表不是普通明细表，经营汇总表取数应来自它的“分类汇总公式结果”。

需要识别这些项目：

```text
学生福利及教具
房租水电
人工
税金
销售费用
办公
交际费
折旧及摊销
管理中心
财务费用
其他
成本费用合计
收入合计
净利润
实际成本
净利润（不含折旧与摊销）
```

判断：

- 能识别分类汇总区：落库为经营汇总可读结构。
- 只能识别普通损益表：不能拿来冒充经营汇总表。

### 3. 检查源表是否有数据

检查：

```sql
SELECT COUNT(*) FROM pl_detail;

SELECT
    period,
    COUNT(*) AS rows,
    COUNT(DISTINCT company_code) AS companies,
    SUM(amount) AS amount
FROM pl_detail
GROUP BY period
ORDER BY period DESC;
```

判断：

- `pl_detail = 0`：经营汇总表不可用，需要先解析收入成本费用表分类汇总结果。
- `pl_detail > 0`：进入期间和筛选检查。

### 4. 检查导入日志是否有对应报表

检查：

```sql
SELECT
    period,
    report_type,
    status,
    COUNT(*) AS batches,
    SUM(success_rows) AS success_rows
FROM import_logs
GROUP BY period, report_type, status
ORDER BY period DESC;
```

判断：

- 如果只有 `income_statement`，说明导入的是损益表，不是经营汇总表主口径数据。
- 如果有“收入成本费用表”但 `pl_detail` 或经营汇总结果仍为空，说明分类汇总区解析或入库映射有问题。

### 5. 检查页面默认期间

经营汇总表当前优先使用：

```text
get_operating_summary_periods()
```

也就是从 `pl_detail` 取期间。

判断：

- `pl_detail` 为空时，页面会退回其他财务期间，但经营汇总源数据仍然为空。
- 页面应该明确提示“请导入收入成本费用明细表”，不能只显示空表。

### 6. 检查分类汇总公式结果是否能展示

有数据后检查这些项目：

```text
学生福利及教具
房租水电
人工
税金
销售费用
办公
交际费
折旧及摊销
管理中心
财务费用
其他
成本费用合计
收入合计
净利润
实际成本
净利润（不含折旧与摊销）
```

重点判断：

- `收入合计` 是否能取到。
- `成本费用合计` 是否能取到。
- `其他` 是否直接来自收入成本费用表分类汇总结果；如果表内没有结果，再按剩余法校验。
- 本年累计不能继续等于本月数。

### 7. 检查页面展示是否可用

页面只保留：

```text
经营摘要表
主体拆分
来源下钻
```

不再检查宽表。

页面可用标准：

- 空数据时说明缺哪张表。
- 有数据时能看到经营摘要。
- 能看到主体拆分。
- 点击费用项能看到来源明细。
- 不显示写死的“已审核”和固定更新时间。

## 四、当前需要修正的点

### P0：先解析收入成本费用表分类汇总结果

当前 `pl_detail` 为空，且收入成本费用表文件存在，经营汇总表不可用的核心问题是分类汇总结果没有被解析入库。

必须做：

1. 新增收入成本费用表专用解析逻辑。
2. 识别分类汇总公式区。
3. 把分类汇总结果落到经营汇总表可读的数据结构。
4. 页面只读取这套结果，不读取普通 `income_statement` 顶替。

### P0：页面空数据提示要更明确

当前不能只显示“暂无数据”。

应改为：

```text
当前期间没有收入成本费用表分类汇总结果。
请先导入或重新解析“收入成本费用表”，系统会读取其中的分类汇总公式结果生成经营汇总表。
```

### P1：加一个可用性诊断区

经营汇总表页面顶部增加诊断：

```text
源表行数
分类汇总结果行数
可用期间
公司数
最近导入批次
缺失原因
```

这样下次页面空了，可以直接看到是没导入、期间不对，还是筛选没命中。

## 六、验收标准

这轮不要求页面一定有金额，因为当前数据库确实没有源数据。

验收只看：

1. 页面能明确告诉用户为什么没数据。
2. 能判断是文件不存在、分类汇总区无法识别、结果未入库，还是筛选没命中。
3. 有分类汇总结果后，经营汇总表能自动显示。
4. 不再用 `income_statement` 假装经营汇总表数据。
