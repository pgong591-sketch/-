# 导入稳定化方案

## 1. 当前目标

本阶段目标是让每月 Excel 上传过程稳定、可追踪、可配置，减少依赖硬编码和人工修复脚本。

本轮先完成公司简称映射配置化，后续继续处理重复导入策略、导入预检和脚本收敛。

## 2. 公司简称映射配置化

### 2.1 新增配置表

新增 `company_aliases` 表，用于维护源文件中的公司简称、别名、文件名简称到正式公司编码的映射。

字段：

| 字段 | 说明 |
| --- | --- |
| id | 主键 |
| alias | 源文件中出现的简称或别名 |
| company_code | 正式公司编码 |
| source | 来源，默认 `manual`，从公司主数据初始化为 `companies` |
| status | 是否启用，1 启用，0 停用 |
| created_at | 创建时间 |
| updated_at | 更新时间 |

### 2.2 当前执行结果

已从 `companies` 表自动初始化别名：

- 公司编码。
- 公司全称。
- 公司简称。

当前已生成 `company_aliases` 122 条。

### 2.3 解析优先级

后续导入时，公司匹配顺序为：

1. 优先查 `company_aliases` 启用别名。
2. 再查 `companies` 的 code、name、short_name 精确匹配。
3. 再查 `companies` 的 name、short_name 模糊匹配。
4. 最后保留旧硬编码映射作为迁移期兜底。

这样新增公司简称时，不需要修改代码。

### 2.4 新增别名方式

示例：

```sql
INSERT OR REPLACE INTO company_aliases
    (alias, company_code, source, status)
VALUES
    ('拔创', '101010136', 'manual', 1);
```

停用别名：

```sql
UPDATE company_aliases
SET status = 0
WHERE alias = '拔创';
```

## 3. 后续任务

### 3.1 重复导入策略

当前系统对同一公司、同一期间、同一报表类型的成功导入会拒绝重复导入。

后续建议改为三种模式：

| 模式 | 说明 | 适用场景 |
| --- | --- | --- |
| reject | 默认拒绝重复导入 | 常规月度导入 |
| replace | 删除旧批次并重新导入 | 财务修正文件 |
| version | 保留多个版本，以最新有效版本出报表 | 审计追溯场景 |

### 3.2 导入预检

建议在真正写入数据库前返回：

- 识别出的公司。
- 识别出的期间。
- 识别出的报表类型。
- 是否存在重复导入。
- 是否有未识别公司别名。
- 是否有字段缺失。

### 3.3 脚本收敛

将 `scripts/` 目录划分为：

- `正式运维脚本`：可长期使用。
- `一次性修复脚本`：已执行、需归档。
- `调试检查脚本`：仅开发使用。

## 4. 本轮已实现的重复导入接口

`import_excel_to_db` 已新增 `duplicate_strategy` 参数：

| 模式 | 当前状态 | 说明 |
| --- | --- | --- |
| reject | 已实现，默认值 | 同公司、同期间、同报表类型已有成功导入时拒绝 |
| replace | 已实现 | 删除旧数据，将旧成功日志标记为已替换，再写入新数据 |
| version | 预留 | 后续用于保留多个版本并按有效版本出报表 |

调用示例：

```python
import_excel_to_db(
    file_path="data/xxx.xlsx",
    company_code="101010136",
    period="202603",
    report_type="资产负债表",
    duplicate_strategy="replace",
)
```
