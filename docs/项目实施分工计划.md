# 财务数据仓库项目实施分工计划

## 1. 项目目标

建设一套面向集团多公司财务数据的轻量级数据仓库和报表生成系统，支持月度 Excel 数据导入、公司层级管理、单体报表、汇总报表、合并口径报表和标准 Excel 导出。

当前系统已经具备可运行原型，包括数据库、解析器、Streamlit 页面、公司层级、部分真实数据和导入日志。后续工作重点从“功能试通”转为“稳定化、配置化、模板化、可交付”。

## 2. 项目组织分工

项目按大公司项目制拆成 8 个板块：

| 板块 | 角色定位 | 主要职责 | 当前优先级 |
| --- | --- | --- | --- |
| PMO/项目管理 | 项目统筹 | 范围、里程碑、风险、验收、变更管理 | P0 |
| 财务业务组 | 业务口径 owner | 公司口径、科目口径、报表口径、合并规则 | P0 |
| 数据架构组 | 数据底座 owner | 数据库 schema、字段字典、批次、历史策略 | P0 |
| 导入解析组 | 数据入口 owner | Excel 识别、解析、校验、入库、错误报告 | P0 |
| 报表模板组 | 最终报表 owner | 三大报表、汇总报表、导出样式、公式配置 | P0 |
| 前端产品组 | 使用体验 owner | Streamlit 页面、导入流程、查询导出流程 | P1 |
| 测试质量组 | 质量 owner | 自动化测试、业务校验、回归测试 | P0 |
| 运维交付组 | 运行保障 owner | 部署、备份、恢复、权限、发布记录 | P1 |

## 3. 当前进度判断

### 3.1 已完成

- 数据库已存在核心表：公司、股权、科目余额、导入日志、模板表、明细表。
- Streamlit 主入口已覆盖导入、查询、三大报表、公司层级、合并汇总、多期查询、后台管理等页面。
- Excel 解析器已覆盖科目余额、资产负债表、损益表、收入人次、管理中心、课酬等多类报表。
- 公司层级管理已实现 tree_path、子树查询和汇总范围获取。
- 当前 SQLite 数据库已有真实业务数据：
  - 公司 60 家。
  - 科目余额 4151 行。
  - 科目余额覆盖 32 家公司。
  - 当前期间为 202603。
  - 导入日志 71 条，状态均为成功。
  - 导入类型包括 balance_sheet、account_balance、income_statement。
- 代码通过 Python 编译检查。

### 3.2 未完成

- pytest 未安装，现有测试套件无法运行。
- README、docs、config 以及部分源码注释存在中文乱码，需要修复或重写。
- 真实数据库结构和 db/init.sql 存在不同步风险。
- 三大报表模板表为空，模板驱动报表尚未真正配置完成。
- 公司简称映射大量硬编码在代码中，后续需要配置化。
- scripts 目录中有大量一次性检查、修复、重导脚本，需要分类和收敛。
- 导入覆盖、重导、版本管理策略还不够正式。

## 6. 风险清单

| 风险 | 影响 | 优先级 | 应对 |
| --- | --- | --- | --- |
| 中文编码乱码 | 影响维护、测试和业务确认 | P0 | 重写关键文档和常量 |
| DB 与 SQL 脚本不同步 | 新环境无法复现 | P0 | 导出现有 schema 并对齐 init.sql |
| 模板表为空 | 三大报表生成不可交付 | P0 | 优先配置最终模板 |
| 公司简称硬编码 | 新公司/新简称需要改代码 | P0 | 改为配置表或配置文件 |
| 重复导入策略不清晰 | 可能覆盖历史数据 | P0 | 建立批次和重导策略 |
| 测试依赖缺失 | 无法回归 | P0 | 补 pytest 并建立测试基线 |
| 一次性脚本过多 | 维护成本高 | P1 | 分类、归档、合并正式逻辑 |
| 内部交易抵消未实现 | 合并报表只能简单汇总 | P1 | 先做一览表，再做自动抵消 |
