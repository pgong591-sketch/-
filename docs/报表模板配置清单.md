## 4. 表格逻辑与核心公式记录规范

后续用户提供的每个表格逻辑、核心公式和取数口径，均先登记在本文件对应模板段落中，再进入数据库模板表或导出逻辑开发。记录时保留用户原始业务表述，同时补充系统可落地口径，避免只留下 Excel 单元格引用而无法追溯业务来源。

### 4.1 公式记录字段

每条公式规则按以下字段记录：

| 字段 | 记录要求 |
| --- | --- |
| 记录编号 | 同一模板内连续编号，如 TPL-002-F001 |
| 模板编号 | 对应模板总清单编号 |
| 表格/工作表名称 | 用户口头描述或 Excel 工作表名称 |
| 行项目 | 报表项目名称、行次、排序号 |
| 公式类型 | 科目范围、SQL表达式、固定值、小计/合计公式 |
| 核心公式 | 用户确认的业务公式，优先保留原始口径 |
| Excel 公式 | 如用户提供单元格公式，原样记录 |
| 系统取数口径 | 后续系统生成时使用的表、字段、过滤条件 |
| 科目范围/条件 | 可落为 JSON、SQL 或明确计算规则 |
| 正负号规则 | 加减方向、借贷方向、亏损填列方式 |
| 校验关系 | 合计、勾稽、期初期末关系等 |
| 待确认点 | 未定死的口径统一标记为待确认 |
| 落地状态 | 待整理、待确认、可配置、已入库、已验证 |

### 4.2 可落地公式类型

- `科目范围`：后续可转换为 JSON，例如 `[{"from":"1001","to":"1012","sign":"+"}]`。
- `SQL表达式`：用于复杂取数，SQL 中默认使用 `:company`、`:period` 作为公司和期间参数。
- `固定值`：用于占位、人工固定金额或尚未确认明细口径的模板行。
- `小计/合计公式`：先完整记录业务公式；如果现有模板表字段无法直接承接，后续开发时再判断扩展模板表、改写 SQL 表达式或在报表引擎中计算。

### 4.3 登记和开发衔接规则

- 每收到一张表或一组公式，先判断归属模板，再写入对应模板段落的“公式记录台账”。
- 可直接映射到现有字段的规则，同时标注目标字段：`formula_type`、`account_ranges`、`sql_expression`、`sign`、`is_subtotal`、`subtotal_group`、`sort_order`。
- Excel 单元格公式必须同时记录“Excel 公式”和“系统取数口径”；如果系统口径暂不清楚，先标为“待确认”。
- 未明确的公式不擅自写死为最终规则，统一进入待确认点。
- 模板规则记录工作独立于导入、数据库恢复和前端样式调整；涉及其他工作流时只在备注中引用，不直接混合实施。

## 5. 模板验收标准

所有最终模板统一按以下标准验收：

- 行次、项目名称、排序与财务确认版一致。
- 取数公式清晰，可追溯到科目余额表或明细表。
- 小计、合计、净额等公式正确。
- 资产负债表满足资产总计 = 负债和所有者权益总计。
- 损益表满足收入、成本、费用、利润之间的勾稽关系。
- 现金流量表满足现金及现金等价物净增加额与期初期末关系。
- 导出 Excel 的表头、单位、期间、公司名称、数字格式符合最终模板。
- 每次模板变更有记录，有测试样例验证。
